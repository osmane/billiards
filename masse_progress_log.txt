Findings:

1. COORDINATE SYSTEM:
   - X, Y: horizontal table plane
   - Z: vertical (up = Vector3(0, 0, 1))
   - Ball position z-coordinate is implicit (always at table height)
   - Current implementation is 2D physics on table surface

2. BALL SPIN REPRESENTATION (src/model/ball.ts):
   - Ball.vel: Vector3 linear velocity (x, y, z components)
   - Ball.rvel: Vector3 angular velocity (rotation around x, y, z axes)
   - Ball.pos: Vector3 position
   - Physics context (physicsContext) contains R (radius) and m (mass)
   - States: Stationary, Rolling, Sliding, Falling, InPocket

3. TABLE CONSTRAINT MECHANISM:
   - CRITICAL FINDING: Ball z-position is IMPLICIT, not enforced
   - Ball.pos has z component but physics appears 2D
   - No explicit constraint keeping ball.pos.z = R or ball.vel.z = 0
   - Ball stays on table through physics model, not hard constraints
   - Need to verify: Is z-velocity ever non-zero?

4. SPIN MECHANICS (src/model/physics/physics.ts):
   - cueToSpin(offset, v): Converts strike offset to angular velocity
   - Current model supports 3D rotation axes (rvel is Vector3)
   - Spin axis from strike: perpendicular to (offset direction, velocity)
   - Formula uses cross products: supports arbitrary rotation axes ✓

5. FORCE APPLICATION (src/model/ball.ts:74-94):
   - updateVelocity() called each frame
   - Two states: Rolling vs Sliding
   - Rolling: forceRoll() then apply rollingFull() delta
   - Sliding: apply sliding() delta
   - Deltas have { v, w } components (velocity & angular velocity changes)
   - Integration: Simple Euler (addScaledVector by timestep)

6. PHYSICS MODELS (src/model/physics/physics.ts):
   - sliding(v, w, context): Returns delta for sliding friction
   - rollingFull(w, context): Returns delta for rolling friction
   - surfaceVelocity(v, w): v + (up × w) * R (contact point velocity)
   - No Magnus force implementation found ✓ (as expected)

7. TIME STEP & INTEGRATION:
   - Table.advance(t) in src/model/table.ts
   - Variable time step with collision detection
   - Retries if collision detected, adjusts timing
   - Ball.update(t): position += velocity * t, then updateVelocity(t)
   - Simple Euler integration method

8. CUE STRIKE MECHANICS (src/view/cue.ts:56-64):
   - Cue angle (aim.angle): rotation around z-axis (horizontal direction)
   - Power (aim.power): magnitude of initial velocity
   - Offset (aim.offset): Vector3 for strike point (x, y from center)
   - Strike: vel = direction * power
   - Then: rvel = cueToSpin(offset, vel)
   - NO cue elevation angle currently (always horizontal strike)

9. PHYSICS CONSTANTS (src/model/physics/constants.ts):
   - R (ball radius): 0.028575m (pool), context-dependent
   - m (ball mass): 0.17kg (pool), context-dependent
   - g = 9.8 m/s²
   - muS = 0.16 (sliding friction)
   - Multiple physics contexts: POOL_PHYSICS, SNOOKER_PHYSICS, CAROM_PHYSICS

10. TEST FRAMEWORK:
    - Jest with TypeScript (test/jest.config.js)
    - Test pattern: describe/it with chai assertions
    - Example: test/model/physics.spec.ts
    - Run: npm test

ARCHITECTURE INSIGHTS:

1. Physics Delta Pattern:
   All physics forces return { v: Vector3, w: Vector3 } delta
   This is the pattern for Magnus force too

2. Ball Update Flow:
   Table.advance(t) → Ball.update(t) → updatePosition(t) + updateVelocity(t)
   Magnus force should be added in updateVelocity() or in sliding()/rolling()

3. Coordinate Convention:
   - Table is in XY plane
   - Z is up
   - Rotation: right-hand rule around axes
   - upCross(v) = (0,0,1) × v for perpendicular vectors

4. Critical Constraint Issue:
   Current code doesn't explicitly enforce ball.vel.z = 0
   Must verify ball never leaves table before adding Magnus force
   May need to add hard constraint: ball.vel.z = 0 always
